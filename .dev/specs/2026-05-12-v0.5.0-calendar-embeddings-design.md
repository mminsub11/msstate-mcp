# v0.5.0 — Calendar Hybrid Retrieval (BM25 + Dense Vectors)

**Status:** design, pre-implementation
**Author:** brainstorming session, 2026-05-12
**Predecessor:** v0.4.1 (`.dev/specs/2026-05-11-v0.4.1-quality-improvements-design.md`)
**Target branch:** `claude/v0.5.0-embeddings`

---

## 1. Problem statement & goal

### Problem

v0.4.1's BM25 retrieval has a hard semantic ceiling on natural-language queries against the calendar corpus. Concrete failures observed against the v0.4.1 921-row corpus:

- `"when does the semester start"` → no high-scoring match for `event="Classes begin"` because the query tokens never appear in the indexed text. v0.4.1's smart-fallback rescues this only when a term is mentioned (`"spring 2027"`); the bare query still fails.
- `"turkey day"` → empty result, even though `event="Thanksgiving holiday"` exists.
- `"deadline to pay tuition"` → BM25 matches via `"Deadline"` token overlap, but ranks weakly when competing with longer event titles.

The v0.4.1 smart-fallback patch only mitigates term-anchored queries. The general semantic-gap class is unresolved.

### Goal

Ship **hybrid BM25 + dense-vector retrieval** in both surfaces (Cloudflare Worker and stdio `npx` plugin) using Voyage `voyage-3-lite` for embeddings, fused via Reciprocal Rank Fusion. Released as **v0.5.0**.

**Success criterion** — measured by `evals/calendar-hybrid-eval.ts`:

- **Semantic-gap bucket (15 queries):** top-3 recall improves **≥ +15 percentage points absolute** vs BM25-only baseline.
- **BM25-favorable bucket (10 queries):** top-3 recall **does not regress more than 5pp** vs BM25-only.
- **Smart-fallback bucket (5 queries):** both modes ≥ 4/5 (validates that the v0.4.1 fallback layer is preserved end-to-end).

### Non-goals

- **Cross-encoder reranking.** Second API hop per query × top-N rows; ~2× latency for marginal recall on a 921-row corpus. Revisit when corpus grows beyond ~10K rows.
- **Policy-side embeddings.** Policy chunks are long PDFs where BM25 already performs well; the semantic gap is calendar-specific. Out of scope; v0.6+ if eval surfaces a need.
- **Multilingual support.** `voyage-3-lite` is English-tuned and the corpus is English-only.

## 2. Architecture overview

Two-stage retrieval that fuses at **rank level**, not score level. RRF avoids the score-normalization problem and is robust to outlier scores.

```
                           ┌──────────────────────────────────────────────┐
                           │ Build time (scripts/build-worker-corpus.mjs) │
                           │  - scrape rows (existing)                    │
                           │  - for each row: POST /v1/embeddings (Voyage)│
                           │  - int8-quantize 512-dim vectors             │
                           │  - bake into worker/corpus.json              │
                           │  - bake sidecar dist/calendar-vectors.json   │
                           │  - hash-skip unchanged rows on rebuild       │
                           └──────────────────────────────────────────────┘
                                             │
                              ships in committed JSON files
                                             ▼
┌─────────────────────┐                ┌──────────────────────────────────┐
│  Worker             │                │  Stdio plugin                    │
│  reads corpus.json  │                │  scrapes live MSU at startup;    │
│  (rows + vectors    │                │  for each row: look up vector by │
│  + BM25 index)      │                │  contentHash in baked sidecar;   │
│                     │                │  cache-miss → embed live + write │
│                     │                │  to ~/.cache/msstate-mcp/embed/  │
│                     │                │  (0o700 dir, 0o600 files)        │
└──────────┬──────────┘                └───────────────┬──────────────────┘
           │                                           │
           │             user query                    │
           ▼                                           ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  find_msu_date handler (mirrored in both surfaces):          │
   │   1. bm25SearchRanked(q, topK=20) → ranked list A            │
   │   2. embedQuery(q) via Voyage REST (LRU 256, 2s timeout)     │
   │      → Float32Array | null                                   │
   │   3. if null → matches = top10 from A, notes += BM25-only    │
   │   4. else: cosineSearchRanked(qVec, topK=20) → ranked list B │
   │   5.       RRF(A, B, k=60) → top 10                          │
   │   6. v0.4.1 smart-fallback layer (unchanged) → +≤3 rows      │
   │   7. return payload with notes describing retrieval mode     │
   └──────────────────────────────────────────────────────────────┘
```

### Key invariants

- **One source of truth for vectors:** the build script writes `worker/corpus.json.academic_calendar.embeddings` and `msstate-policies/dist/calendar-vectors.json` from the same in-memory map in a single pass. They are byte-identical for the same `contentHash` set.
- **Two retrieval paths, one fusion:** BM25 and cosine ranks merged via RRF with canonical Cormack et al. `k=60`. Worker and stdio use the same constant.
- **Graceful degradation is first-class:** missing key, embed call failure, or tripped circuit breaker → BM25-only path with `notes` field telling the LLM the result is degraded. No silent quality drop.
- **API key boundary:** Worker reads `env.VOYAGE_API_KEY` from wrangler secrets. Stdio reads `process.env.VOYAGE_API_KEY`. Never logged, never echoed in client errors, never sent anywhere except `api.voyageai.com`.

## 4. Data flow

### Build time (`scripts/build-worker-corpus.mjs`)

```
1. scrape all 6 sources → rows[]                      (existing v0.4.1 logic)
2. for each row: contentHash = sha256(`${event}|${term||""}|${description||""}`)
3. read existing worker/corpus.json → existingVecMap = { hash → {scale, data_b64} }
4. newRows = rows.filter(r => !existingVecMap[r.hash])
5. for each newRow: embedText(r) =
     `${event}${term?" — "+term:""}${description?". "+description:""}`.slice(0, 1024)
6. batched POST https://api.voyageai.com/v1/embeddings
     body: { model: "voyage-3-lite", input: [...], input_type: "document" }
     (128 per batch, retry 1s/3s/10s on 429/5xx, abort on >10% per-batch failure)
7. quantize each Float32Array → { scale, data: Int8Array } → base64
8. mergedVecMap = { ...existingVecMap, ...newVecMap }
9. write worker/corpus.json (rows + embeddings + embed_model + embed_dim)
10. write msstate-policies/dist/calendar-vectors.json (hash→{scale,data_b64} only)
11. abort if: per-source row count is 0 || >10% embed failure || vector count != row count
    → throw with canonical string "refusing to ship a poisoned calendar corpus"
```

**Re-embed cost (clean rebuild):** 921 rows × ~30 input tokens ≈ 28K tokens. Voyage free tier is 50M/month, so each clean rebuild consumes 0.056% of monthly quota. Incremental rebuilds (hash-skip) typically embed < 50 changed rows.

### Query time (mirrored in both surfaces)

```
input.q (e.g. "when does the semester start spring 2027")
  │
  ├─→ bm25SearchRanked(q, topK=20) → [(rowId, rank), …]    (always runs)
  │
  └─→ embedQuery(q) → Float32Array | null                  (may degrade)
        │
        ├── null (no key / call failed / circuit-breaker tripped):
        │    matches = top10 from BM25 only
        │    notes += "embedding rerank unavailable; using BM25 only"
        │
        └── Float32Array:
             cosineSearchRanked(qVec, vecMap, topK=20)
               → for each indexed row with a vector:
                   score = cosine(qVec, dequantize(vecMap[row.hash]))
               → sort desc → take top 20
             matches = RRF(bm25_ranks, cosine_ranks, k=60).slice(0, 10)
             notes += "hybrid retrieval (BM25 + voyage-3-lite)"

  matches → v0.4.1 smart-fallback layer (unchanged)
           → if term mentioned AND <2 academic_calendar rows in matches,
             append ≤3 academic_calendar rows tagged fallback:true
  → buildNotes(matches, fallbackTriggered, term, embeddingMode)
  → JSON-RPC response
```

**RRF formula:** `score(d) = Σ_L 1 / (k + rank_L(d))` over lists `L`. `k=60` is canonical and shared between surfaces.

### Latency budget (Worker, p50)

| Step | Cost |
|---|---|
| BM25 | < 2ms (in-memory, 921 rows) |
| `embedQuery` (Voyage call, LRU miss) | 80–120ms |
| `embedQuery` (LRU hit, ~30% of repeat queries) | < 3ms |
| Cosine + RRF | < 1ms |
| **Total additional vs v0.4.1 BM25-only** | **~100ms p50 on cache miss** |

Acceptable per design discussion.

## 5. Error handling & graceful degradation

### F1 — `VOYAGE_API_KEY` unset

- **Worker:** `env.VOYAGE_API_KEY` undefined at module load → set `isEmbeddingAvailable = false`. Log once via `console.error` (stderr-equivalent): `"VOYAGE_API_KEY unset; calendar tool falling back to BM25-only retrieval"`. No crash. Every `find_msu_date` call skips the embed step.
- **Stdio:** identical check on `process.env.VOYAGE_API_KEY`. `embed.ts` exports `isEmbeddingAvailable` constant.
- Response `notes`: `"embedding rerank unavailable; using BM25 only"`.

### F2 — Voyage API call fails at query time

- Wrap `embedQuery` in 2s timeout (`AbortSignal.timeout(2000)`).
- On error: log structured `{ err_class, duration_ms, status }` server-side. **Never echo `err.message` to client** (matches the round-2 audit closure pattern N3). Return `null` → caller falls back to BM25.
- **No retries at query time** — 2s tail is already long; better to degrade than make the user wait longer.
- **Circuit breaker:** 5 consecutive failures within a 30s sliding window → flip `embeddingMode = "tripped"` for 60s. Skip Voyage calls during the trip window. Per-isolate state on Worker (independent trips per isolate is fine — they recover independently too).

### F3 — Build-time embedding failures

- Per-batch retry with exponential backoff: 1s / 3s / 10s.
- If a batch still fails after retries → **abort the whole build** with `"refusing to ship a poisoned calendar corpus"` (canonical string from v0.4.1 build guard). Partial vector sets are invisible bit-rot; better to fail loudly.
- If >10% of rows fail across all batches → abort.
- **Model-change protection:** if existing `worker/corpus.json` has `embed_model !== "voyage-3-lite"`, build aborts unless `--force-reembed` flag is passed. Old vectors are incompatible with new model embeddings.

### F4 — Stdio plugin cache-miss + offline

- If stdio scrapes a row whose `contentHash` is not in the sidecar **and** the live embed call fails: row gets indexed without a vector. BM25 still scores it. At query time, cosine search simply skips rows without vectors — no error, just a slightly degraded ranking for the missing row. Logged once per row at warn level.

### `notes` field additions

The v0.4.1 `notes` field (currently used for fallback explanation and multi-year disambiguation) gets two new possible suffixes appended in order:

- Always: one of `"hybrid retrieval (BM25 + voyage-3-lite)"` OR `"embedding rerank unavailable; using BM25 only"` (depending on mode for this query).
- If smart-fallback fired: existing v0.4.1 fallback notes (unchanged).

Stays terse. Default queries produce a 1-line notes value.

## 6. Eval methodology

`msstate-policies/evals/calendar-hybrid-eval.ts` ships a held-out 30-query eval set.

### Query buckets

1. **Semantic-gap (15 queries)** — paraphrases / colloquialisms BM25 fails on. Embeddings *should* fix these.
   - Example: `"when does the semester start"` → expect `Classes begin` rows.
   - Example: `"turkey day"` → expect `Thanksgiving holiday` rows.
   - Example: `"deadline to pay tuition"` → expect `Deadline for payment in full…` rows.

2. **BM25-favorable (10 queries)** — exact-term queries BM25 already handles well. Guards against hybrid regressing easy cases.
   - Example: `"final examinations spring 2026"` → expect `Final Examinations` rows.
   - Example: `"commencement december 2026"` → expect `Commencement` rows in Fall 2026.

3. **Smart-fallback (5 queries)** — term mentioned, non-academic primary source. Guards v0.4.1 fallback layer survives hybrid retrieval.
   - Example: `"grad student spring 2027 schedule"` → expect academic_calendar rows via fallback.

### Corpus-rule compliance

- Queries are **synthesized** in the brainstorming session (allowed — they're test inputs, not corpus content).
- **Expected event titles** are pulled from real rows in `worker/corpus.json` at eval-write time. Every expected event comes from a verified live MSU row — none authored from memory.
- Eval file records the `corpus_built_at` it was authored against, so a future maintainer can tell when ground truth was last verified.

### Metrics

Per query: top-3 recall = 1 if any expected event title appears in matches[0..2], else 0.

Per bucket aggregate:

| Bucket | Target |
|---|---|
| Semantic-gap | hybrid recall@3 ≥ BM25-only recall@3 + **15pp** |
| BM25-favorable | hybrid recall@3 ≥ BM25-only recall@3 − **5pp** |
| Smart-fallback | both modes ≥ 4/5 |

### Where eval runs

- **Local:** `npm run eval:hybrid` runs both modes (forces BM25-only by spawning a subprocess with `VOYAGE_API_KEY=` blanked; runs hybrid with key set), prints a delta table.
- **CI:** runs BM25-only mode only (no Voyage key in CI secrets — out of scope to add for v0.5.0). Verifies the eval file parses and reproduces the v0.4.1 baseline at minimum. Hybrid mode runs locally before each release tag.
- **GitHub Action with live key:** deferred to v0.5.1.

### Failing-the-build conditions

- Hybrid bucket-1 lift < +10pp → ship-blocker (target was +15pp; <10pp means embeddings didn't earn the dependency).
- Hybrid bucket-2 regression > 5pp → ship-blocker (broke the easy queries).
- Eval file does not load or expected events don't appear in current corpus → ship-blocker (stale eval ground truth).

### Cost note

Weekly eval cost: 30 queries × ~15 tokens ≈ 450 tokens/week ≈ 1,800 tokens/month. On Voyage's 50M-token free tier this is 0.0036% of quota; on the paid tier it's $0.000036/month. Operational rounding error.

### Threat model additions to `SECURITY.md`

- **New egress:** user query text leaves the runtime to `api.voyageai.com`. Documented as in-scope egress. Query text is bounded at 4096 chars by `MAX_QUERY_CHARS`. Voyage's privacy policy (no training on API inputs) is referenced.
- **API key compromise:** scoped — Voyage key can only spend the embedding quota; cannot read the policy corpus, cannot pivot to MSU systems. Rotation procedure documented: `wrangler secret put VOYAGE_API_KEY` for the Worker; npm-publish users self-rotate.
- **Out of scope (unchanged from v0.4.1):** local edits to the published bundle, prompt-level circumvention by the LLM client, fork-the-corpus, indirect prompt injection in MSU-published PDFs.

## 8. Out of scope (v0.5.0) and future work

Explicitly **not** in v0.5.0. Listed so the spec sets expectations and a future maintainer doesn't think these were forgotten.

### Deferred to later versions

- **Cross-encoder reranking** — second API hop per query × top-N rows; ~2× latency for marginal recall gain on a 921-row corpus. Revisit if corpus grows past ~10K rows. v0.6+.
- **Multilingual / Spanish query support** — `voyage-3-lite` is English-tuned and the corpus is English-only. v0.6+ if demand surfaces and source content gains translations.
- **CI eval with live Voyage key** — needs a repo secret + per-run budget. v0.5.1 will add a scheduled GitHub Action with the key wired via repo secret. v0.5.0 CI runs BM25-only mode + local-only hybrid bench before release tagging.
- **Policy-side embeddings** — separate retrieval shape (long PDF chunks, BM25 already strong). v0.6+ if eval reveals a gap.
- **Adaptive RRF `k`** — per-query learned `k` is research-grade. Fixed `k=60` is the standard.
- **User-facing "why this match" payload** — surface BM25-rank + cosine-rank contribution per match so the LLM can explain rank decisions. LLM does fine without it.

### Quota monitoring (open question to revisit at release)

Voyage free tier is 50M tokens/mo. At ~30 tokens/query × 1M queries/mo = 30M. End-user query volume would need to exceed ~1.6M queries/month to break the free tier — not realistic for an MSU policies tool, but documented here so a future maintainer can spot the cliff. Adding alerting is out of scope for v0.5.0.

### Versioning roadmap

- **v0.5.0** (this spec): hybrid retrieval for calendars, both surfaces, BM25-graceful fallback, security checklist 192 → 222.
- **v0.5.1**: CI eval with live key + scheduled weekly bench; minor tuning based on production query patterns.
- **v0.6.0**: policy-side embeddings if eval reveals a gap; cross-encoder rerank if corpus grows.

## Appendix A — Worked example

**Query:** `"when does the semester start in spring 2027"`

### v0.4.1 (BM25-only)

1. BM25 tokenizes → `[when, does, the, semester, start, in, spring, 2027]`.
2. `semester` and `start` have low IDF (rare in corpus); `spring` and `2027` match many rows.
3. Top-3 results are likely `Spring Break, Spring 2027`, `Spring 2027 application deadline`, `Last day to drop, Spring 2027`. **None of which answer the user's question.**
4. Smart-fallback: term `Spring 2027` detected, <2 academic_calendar rows in matches → appends 3 fallback rows including `Classes begin, Spring 2027`. **Question answered, but via fallback heuristic, not retrieval.**

### v0.5.0 (hybrid)

1. BM25 produces the same noisy ranking as above.
2. `embedQuery("when does the semester start in spring 2027")` → 512-dim vector.
3. Cosine similarity against baked row vectors: `Classes begin, Spring 2027` ranks #1 (semantic match on "semester start" → "classes begin"). `Spring Break` ranks lower because "break" is semantically distant from "start".
4. RRF fuses BM25 ranks (where `Spring Break` was #1) and cosine ranks (where `Classes begin` is #1) → final top-3 includes `Classes begin` as #1 or #2.
5. Smart-fallback is still applied but is rarely needed because retrieval already produced the right answer.

**Outcome:** the right row is found via *retrieval*, not via heuristic fallback. Fallback remains as a backstop for the long tail.


### `worker/corpus.json` (new fields on `academic_calendar`)

```json
{
  "academic_calendar": {
    "rows": [...],

    "built_at": "...",

      "a3f2…hash": { "scale": 0.0123, "data": "base64..." }
```

### Voyage REST request

```http
POST https://api.voyageai.com/v1/embeddings
Authorization: Bearer ${VOYAGE_API_KEY}
Content-Type: application/json

{
  "model": "voyage-3-lite",
  "input": ["Classes begin — Spring 2026", "Spring Break — Spring 2026"],
  "input_type": "document"
}
```

Response: `{ "data": [{ "embedding": [0.012, -0.034] }], "usage": {...} }`. At query time `input_type` is `"query"` instead of `"document"`.
