# v0.5.0 — Calendar BM25 with LLM-Generated Synonyms

**Status:** design, pre-implementation (pivoted 2026-05-12 from the previous "hybrid dense-vector retrieval" design)
**Author:** brainstorming session, 2026-05-12
**Predecessor:** v0.4.1 (`.dev/specs/2026-05-11-v0.4.1-quality-improvements-design.md`)
**Target branch:** `claude/v0.5.0-embeddings` (branch name kept for continuity; design has changed)

> **Pivot note:** an earlier version of this spec called for query-time Voyage embeddings. That design was rejected on a hard project constraint: **zero ongoing operational cost** — runtime third-party API dependencies are out of scope. This revision moves all ML work to build time (LLM-generated paraphrases) so the runtime stays pure BM25 with no API egress. The git log shows the earlier draft committed as `a6474cb` and superseded here.

---

## 1. Problem statement & goal

### Problem

v0.4.1's BM25 retrieval has a hard semantic ceiling on natural-language queries. Concrete failures observed against the v0.4.1 921-row corpus:

- `"when does the semester start"` → no high-scoring match for `event="Classes begin"` because the query tokens never appear in the indexed text.
- `"turkey day"` → empty result, even though `event="Thanksgiving holiday"` exists.
- `"graduation ceremony"` → ranks `"Commencement"` weakly because no token overlap.

The v0.4.1 smart-fallback patch only mitigates term-anchored queries. The general semantic-gap class is unresolved.

### Goal

Ship **BM25 with LLM-generated synonyms baked into each row at build time**. At query time the system stays pure BM25 — no runtime API, no vendors, no egress. Released as **v0.5.0**.

**Success criterion** — measured by `evals/calendar-synonyms-eval.ts`:

- **Semantic-gap bucket (15 queries):** top-3 recall improves **≥ +15 percentage points absolute** vs v0.4.1 baseline (BM25 without synonyms).
- **BM25-favorable bucket (10 queries):** top-3 recall **does not regress more than 5pp** vs baseline.
- **Smart-fallback bucket (5 queries):** both modes ≥ 4/5 (the v0.4.1 fallback layer is preserved end-to-end).

### Non-goals

- **Query-time embeddings / vector search.** Adds runtime cost and a third-party dependency. Rejected on the zero-runtime-cost constraint.
- **Cross-encoder rerank.** Same reason.
- **Multilingual / Spanish synonym generation.** The LLM is English-tuned and corpus is English-only.
- **Synonym generation at scrape time.** All LLM work happens in the build script, never in the live-scrape path.

## 2. Architecture overview

Pure BM25 with a multi-field weighted score. The only ML work is the **build-time synonym expansion**.

```
                              ┌──────────────────────────────────────────────┐
                              │ Build time (scripts/build-worker-corpus.mjs) │
                              │  - scrape rows (existing)                    │
                              │  - for each row missing synonyms in existing │
                              │    corpus.json: POST to Anthropic Haiku      │
                              │  - parse 5 paraphrases per row               │
                              │  - bake into row.synonyms: string[]          │
                              │  - hash-skip unchanged rows on rebuild       │
                              │  - abort on >10% paraphrase failure          │
                              └──────────────────────────────────────────────┘
                                                  │
                              ships in committed worker/corpus.json
                              + dist/calendar-synonyms.json (sidecar)
                                                  ▼
┌─────────────────────┐                ┌──────────────────────────────────┐
│  Worker             │                │  Stdio plugin                    │
│  reads corpus.json  │                │  scrapes live MSU at startup,    │
│  with rows that     │                │  then attaches baked synonyms    │
│  already include    │                │  from dist/calendar-synonyms.json│
│  synonyms[]         │                │  sidecar keyed by contentHash    │
└──────────┬──────────┘                └───────────────┬──────────────────┘
           │                                           │
           │             user query                    │
           ▼                                           ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  find_msu_date handler (mirrored in both surfaces):          │
   │    1. BM25 with field weights:                               │
   │         event(3) + synonyms(2) + term(1) + description(1)    │
   │    2. Top 10 → v0.4.1 smart-fallback layer (unchanged)       │
   │       → if term mentioned AND <2 academic_calendar rows,     │
   │         append ≤3 academic_calendar rows tagged fallback     │
   │    3. Strip contentHash from response                        │
   │    4. Notes field describes mode (always "BM25 + synonyms")  │
   └──────────────────────────────────────────────────────────────┘
```

### Key invariants

- **Zero runtime API.** The Worker and the stdio plugin never call any third-party service to serve a query. `fetch` to Anthropic only happens in the build script.
- **Build-time LLM is the only ML.** Anthropic Claude Haiku (`claude-haiku-4-5`) generates ~5 paraphrases per row at build time. Build cost: ~$0.50 per full rebuild for ~921 rows. Cached by `contentHash` so incremental builds re-paraphrase only changed rows.
- **One source of truth for synonyms.** The build script writes `worker/corpus.json` with `row.synonyms` populated AND `msstate-policies/dist/calendar-synonyms.json` (sidecar keyed by `contentHash`) for the stdio plugin to attach to live-scraped rows.
- **`contentHash` field on each row** (preserved from Task 2 work): SHA-256 of `${event}|${term??""}|${description??""}`. Stripped from `find_msu_date` JSON-RPC responses so it doesn't appear in the wire shape.

## 4. Data flow

### Build time (`scripts/build-worker-corpus.mjs`)

```
1. scrape all 6 sources → rows[]                       (existing v0.4.1 logic)
2. for each row: contentHash = sha256(`${event}|${term||""}|${description||""}`)
3. read existing worker/corpus.json → existingSynMap = { hash → string[] }
4. newRows = rows.filter(r => !existingSynMap[r.hash])
5. for each newRow:
     POST https://api.anthropic.com/v1/messages
       model: "claude-haiku-4-5"
       max_tokens: 200
       system: "You generate 5 short paraphrases of MSU calendar event titles
                for keyword-based search. Each paraphrase must preserve the
                semantic meaning, use 1-6 common English words, contain NO
                dates/years/numbers, and be ≤80 characters. Output ONLY a
                JSON array of strings, no preamble, no explanation."
       user: `Event: "${event}"${term ? ` (term: ${term})` : ""}.
              Return 5 paraphrases as a JSON array.`
6. parse + validate response:
     - JSON.parse the assistant content
     - assert isArray && length === 5
     - each item is a string, ≤80 chars, no digits, non-empty
     - on any failure: log and count toward 10% threshold
7. merge: synMap = { ...existingSynMap, ...newSynMap }
8. attach synonyms to rows: for each r, r.synonyms = synMap[r.contentHash] ?? []
9. write worker/corpus.json (rows with synonyms; NO embeddings)
10. write msstate-policies/dist/calendar-synonyms.json (hash → string[])
11. abort if:
      - per-source row count is 0
      - >10% paraphrase failure rate
      - any row has both contentHash set AND synonyms missing from synMap (consistency)
    → throw "refusing to ship a poisoned calendar corpus"
```

**Per-call constraint:** one request per row, run with `Promise.all` chunks of 8. ~921 rows / 8 chunks × ~3-5s per call ≈ **4-8 min wall time** for a clean rebuild. Incremental rebuilds usually <30s.

**Re-paraphrase cost (clean rebuild):**
- Input: ~50 tokens × 921 rows = ~46K input tokens × $0.80/1M = **$0.04**
- Output: ~100 tokens × 921 rows = ~92K output tokens × $4/1M = **$0.37**
- **Total: ~$0.41 per full rebuild.** Incremental rebuilds typically <$0.05.

### Query time (mirrored in both surfaces)

```
input.q (e.g. "when does the semester start spring 2027")
  │
  ▼
bm25Search(q, limit=10) with field weights
  event(3) + synonyms(2) + term(1) + description(1)
  → top 10 hits

  matches → v0.4.1 smart-fallback layer (unchanged)
           → if term mentioned AND <2 academic_calendar rows in matches,
             append ≤3 academic_calendar rows tagged fallback:true

  → strip row.contentHash from each wire match
  → buildNotes(matches, fallbackTriggered, term)
       always includes "BM25 with synonyms" — there's no degradation path
       to gate on; corpus is either built or not.
  → JSON-RPC response
```

**Latency** (Worker, p50): same as v0.4.1 BM25 — under 5ms. No regression.

## 5. Error handling

### F1 — Build-time `ANTHROPIC_API_KEY` unset

- Build script errors fast: `throw new Error("ANTHROPIC_API_KEY is required for the build step. Export it and re-run.")` before any scrape happens.
- Build aborts. No partial output. Consistent with the v0.4.1 build-abort pattern.

### F2 — Anthropic API call fails for a single row

- Per-row retry: 1s, 3s, 10s exponential backoff (max 3 retries).
- On final failure: count toward the 10% failure threshold. Don't crash — accumulate failures and decide at the end.
- If total failures exceed 10% of newRows: throw `"refusing to ship a poisoned calendar corpus"`.

### F3 — Anthropic response malformed

- If response is not valid JSON, not an array, not 5 items, contains digits, or exceeds 80 chars per item: log and treat as a per-row failure (same threshold as F2).
- Never silently accept a bad paraphrase set — the corpus rule requires real, validated data.

### F4 — No runtime failure mode

- Because there's no runtime API, the system has nothing to fail in production query handling beyond what v0.4.1 already handles (BM25 index corruption, etc.). No new failure modes to test for at query time.

### `notes` field

The v0.4.1 `notes` field gains one suffix: `"BM25 with synonyms"`. It's appended after any existing v0.4.1 messages (multi-year, smart-fallback). Stays terse.

## 6. Eval methodology

`msstate-policies/evals/calendar-synonyms-eval.ts` ships a held-out 30-query eval set. The 30 queries and their expected event-title prefixes are validated against the live corpus at eval-write time.

### Query buckets

1. **Semantic-gap (15 queries)** — paraphrases / colloquialisms BM25 fails on without synonyms.
2. **BM25-favorable (10 queries)** — exact-term queries BM25 already handles well. Guards against synonyms confusing the easy cases.
3. **Smart-fallback (5 queries)** — term mentioned, non-academic primary source. Guards v0.4.1 fallback layer.

### Eval modes

- **Baseline:** load the corpus but strip every row's `synonyms` field before indexing. Equivalent to v0.4.1 retrieval.
- **Synonyms:** load and index normally.

Both modes run against the same live corpus loaded by `loadAllCalendarRows()`.

### Ship-blocker gates

| Bucket | Gate |
|---|---|
| Semantic-gap | synonyms recall@3 ≥ baseline + **15pp** |
| BM25-favorable | synonyms recall@3 ≥ baseline − **5pp** |
| Smart-fallback | both modes ≥ 4/5 |

### Cost note

The eval makes **zero API calls** — both modes are pure BM25 over the in-memory index. Local-only. CI eval cost is exactly zero.

## 7. Security checklist additions

The round-2 audit closure left `tools/security-checklist.sh` at 192/192 with CI hard-gating at ≥100. v0.5.0's only API surface is at build time (Anthropic). New checks reflect that:

| ID | Check | Pts |
|---|---|---|
| **SYN1** | No `ANTHROPIC_API_KEY` value committed. `grep -rE "sk-ant-[a-zA-Z0-9_-]{20,}" .` returns no matches outside `.gitignore` allowlist. | 5 |
| **SYN2** | Build script aborts on partial paraphrase failure with the canonical `"refusing to ship a poisoned calendar corpus"` string. | 5 |
| **SYN3** | Build script aborts when `ANTHROPIC_API_KEY` is unset, before any scrape. | 3 |
| **SYN4** | **No runtime egress to Anthropic.** `grep -rn "api\.anthropic\.com" msstate-policies/src worker/src \| wc -l` must equal **0**. The only allowed reference is in `scripts/build-worker-corpus.mjs`. | **10** (high — this is the zero-runtime-cost guarantee) |
| **SYN5** | Synonym validation in build script — explicit regex/length checks visible in source. `grep -E "(\.length > 80\|/\\d/)" scripts/build-worker-corpus.mjs` returns matches. | 3 |
| **SYN6** | `row.synonyms` is stripped from any user-facing path in tools (defensive — guard against future regressions): `grep -rn "synonyms" msstate-policies/src/tools \| wc -l` must equal 0. | 2 |
| **CAL5** | (Regression-guard) Worker CORS allowlist still excludes `Authorization`. Carries over from v0.4.1. | 0 |

**New target:** 192 + 28 = **220**. CI threshold stays at 100.

### Threat model additions to `SECURITY.md`

- **Build-time egress.** During corpus rebuild, the build script transits each row's `event` title (and optionally `term`) to `https://api.anthropic.com/v1/messages`. Row data is public MSU calendar entries — no PII. Documented and in-scope.
- **No runtime egress.** Production deployments (Worker, npm) make **no third-party API calls** to serve queries. This is a load-bearing v0.5.0 guarantee enforced by SYN4.
- **ANTHROPIC_API_KEY scope.** Compromise lets an attacker spend the operator's Anthropic quota. Cannot read the corpus, cannot pivot to MSU, cannot impersonate the tool at runtime. Rotation: change the env var, re-run build (no deploy needed if synonyms unchanged).
- **Out of scope (unchanged from v0.4.1):** local edits to the published bundle, prompt-level circumvention by the LLM client, fork-the-corpus, indirect prompt injection in MSU-published PDFs.

## 8. Out of scope (v0.5.0) and future work

### Deferred

- **Runtime embeddings / hybrid retrieval.** The rejected previous design. Reconsider only if (a) eval reveals synonyms can't close the gap and (b) the operational-cost constraint changes.
- **Multilingual synonym generation.** Source corpus is English-only.
- **Cross-encoder rerank.** Out of scope for this corpus size.
- **Policy-side synonyms.** This spec only covers calendar rows. Policy chunks are already long enough that BM25 works well; revisit only if eval surfaces a need.
- **Auto-refreshed paraphrases.** Synonyms might subtly drift in quality if Claude Haiku is updated. Out of scope to detect; a fresh full rebuild (every release) re-paraphrases everything and naturally absorbs model improvements.
- **User-facing "why this match" payload.** Same deferral as before.

### Versioning roadmap

- **v0.5.0** (this spec): BM25 with build-time synonym expansion. Zero runtime cost. Security checklist 192 → 220.
- **v0.5.1**: minor tuning based on production query patterns; possibly more synonyms per row if eval shows headroom.
- **v0.6.0**: re-evaluate runtime embeddings only if the cost constraint changes.

## Appendix A — Worked example

**Query:** `"when does the semester start"`

### v0.4.1 (no synonyms)
1. BM25 tokens: `[when, does, the, semester, start]`.
2. `semester` and `start` have low IDF; no high-scoring row contains them.
3. Top results are dominated by unrelated noise. The smart-fallback only fires if a term is mentioned. **Question unanswered.**

### v0.5.0 (with synonyms)
1. The row for `event="Classes begin", term="Spring 2026"` has `synonyms = ["semester starts", "school begins", "first day of class", "term begins", "instruction begins"]` (baked at build time).
2. BM25 scores the `synonyms` field at weight 2. Query tokens `semester` and `start` match `"semester starts"`.
3. The row ranks #1 or #2. **Question answered via retrieval, no fallback needed.**
