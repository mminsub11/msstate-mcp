**Status:** design, pre-implementation
**Author:** brainstorming session, 2026-05-12
**Predecessor:** v0.5.0 (`.dev/specs/2026-05-12-v0.5.0-calendar-embeddings-design.md`)
**Target branch:** `claude/v0.6.0-courses`

## 1. Problem statement & goal

### Problem

MSU students answering "can I take CSE 4733 next semester?" or "what does CSE 4153 cover?" today must navigate `catalog.msstate.edu` by hand. The catalog renders well in a browser but is hostile to fast questions:

- **Multi-chain prereqs are buried in prose.** The page for CSE 4733 says `(Prerequisites: Grade of C or better in CSE 3724 …)`, and 3724 in turn has its own prereqs, and so on. Walking that chain takes 4–6 page loads per question.
- **Course explainers require finding the dept page first.** There's no top-level "what is course X?" affordance; students hunt through college → department → in-page anchor.
- **Reverse navigation does not exist.** "What unlocks once I finish CSE 1384?" is unanswerable from the catalog UI — there's no inverse index.
- **MCP-capable clients** (Claude Code, Claude Desktop, claude.ai connector, ChatGPT connector) have no MSU-grounded answer source for these questions today. The MCP server already covers policies and academic dates; courses are the third leg.

### Goal

Ship three new MCP tools that answer course-level questions using **only** content scraped from `catalog.msstate.edu`, baked into the existing corpus pipeline, with zero ongoing operational cost. Released as **v0.6.0**.

**Success criterion** — measured by `evals/courses.jsonl`:

- **Course-explainer bucket (~20 queries):** top-1 recall on course code ≥ 90%.
- **Forward prereq-chain bucket (~15 queries):** for each query, the returned chain contains the full expected subset of course codes ≥ 95% of the time.
- **Reverse-unlock bucket (~15 queries):** for each query, the returned set contains the expected subset of unlocked codes ≥ 95% of the time.

### Non-goals

- **Degree-program structure** ("what courses are in BS CSE?"). Parses `sc_courselist` tables with `orclass` alternatives; different data model, larger scope. Deferred to v0.7+ if eval surfaces demand.
- **Archived catalog editions.** `/archives/` covers prior years for students under older requirements. Real value but tripled corpus size and slightly different HTML. v1 ships current undergraduate + graduate only.
- **Embeddings for course search.** BM25 over `code + title + description` is strong on this corpus (descriptive titles, stable dept codes). Adding Voyage or Workers-AI embeddings later is a non-breaking change. Default v0.6.0 path is BM25-only, zero ongoing cost.
- **Section / seat / instructor / textbook data.** None of this lives in `catalog.msstate.edu`; the registrar's BANNER system owns it. Out of corpus scope.
- **"How hard is X?" / "best professor for X" / RateMyProfessor-style answers.** Not in the corpus. Corpus rule (CLAUDE.md) forbids importing.
- **Transfer-credit equivalency** ("does my community college's CS-101 satisfy CSE 1284?"). Requires registrar data we don't have.

## 2. Architecture overview

Same baked-corpus pattern as policies (v0.1–v0.4) and calendars (v0.4). Build-time scraper crawls `catalog.msstate.edu`, parses every course detail page, emits a JSON block into `worker/corpus.json` and a sidecar in `msstate-policies/dist/`. Both surfaces (Cloudflare Worker, stdio `npx` plugin) load the same baked data — byte-identical for the same scrape pass.

```
                           ┌────────────────────────────────────────────┐
                           │ Build time (scripts/build-worker-corpus.mjs)│
                           │  - enumerate depts from /azindex/          │
                           │  - per dept: extract bubblelink course codes│
                           │  - per course: fetch /search/?P=<code>     │
                           │  - parse → Course record + prereq tree     │
                           │  - build forward + reverse DAG             │
                           │  - bake into worker/corpus.json            │
                           └────────────────────────────────────────────┘
                                       │
        ┌──────────────────────────────┼──────────────────────────────┐
        ▼                              ▼                              ▼
┌──────────────────┐         ┌──────────────────┐         ┌──────────────────┐
│ msstate-policies │         │  worker/corpus   │         │  evals/          │
│  /dist/index.js  │         │  .json           │         │  courses.jsonl   │
│  (stdio + plugin)│         │  (CF Worker)     │         │                  │
└──────────────────┘         └──────────────────┘         └──────────────────┘
        │                              │                              │
        ▼                              ▼                              ▼
┌──────────────────────────────────────────────────────────────────────────┐
│ Tool calls (MCP JSON-RPC, identical schemas across surfaces)             │
│   search_msu_courses(q, limit?)                                          │
│   get_msu_course(code)                                                   │
│   get_msu_course_graph(code, direction, depth?)                          │
└──────────────────────────────────────────────────────────────────────────┘
```

### Key invariants

- **Corpus rule (CLAUDE.md).** Course content comes exclusively from `catalog.msstate.edu` and pages it links to. No training data, no third-party mirrors, no `WebSearch`. Eval question authoring requires opening the relevant catalog page in the same session.
- **One source of truth for course records.** The build script writes both `worker/corpus.json.courses` and a sidecar (`msstate-policies/dist/courses.json` — only if size makes splitting worthwhile, else inline in `index.js`) from a single in-memory map in a single pass.
- **Stderr-only logging.** All runtime logs through `src/log.ts`. The build-time scraper subprocess (`scripts/_scrape-catalog.ts`) patches `console.log → stderr` before any imports, mirroring `scripts/_scrape-calendars.ts` (CLAUDE.md note about pdf-parse).
- **Build aborts on poisoned corpus.** Any of: WAF interstitial, HTTP 5xx, zero courses in a dept, prereq-parse exception rate > 5%, or empty top-level corpus → abort with marker `"refusing to ship a poisoned course corpus"`. Same defense pattern as the calendar build added in v0.4.0.
- **Allowlist of catalog URL roots.** Hardcoded in `src/courses/types.ts`. Per-course URLs are extracted from authenticated index/department HTML, never constructed from external input.
- **Zero ongoing operational cost.** No paid APIs at runtime, no cron, no per-query egress to third parties. The tool only spends when a developer triggers a build.

### Tool schemas

**`search_msu_courses`**

```ts
input: { q: string, limit?: number }   // limit default 10, max 50
output: {
  matches: Array<{
    code: string,        // "CSE 4153"
    title: string,
    hours: number | string,   // can be "0,4" or "3-4" per catalog
    dept: string,        // "CSE"
    level: "undergraduate" | "graduate",
    score: number        // BM25
  }>,
  notes: string[]        // diagnostics, never leak internal state
}
```

**`get_msu_course`**

```ts
input: { code: string }   // validated /^[A-Z]{2,4}\s\d{4}$/
output:
  | { found: true, course: Course }
  | { found: false, code: string, suggestions: Array<{code, title}> }

Course = {
  code: string,
  title: string,
  hours: number | string,
  level: "undergraduate" | "graduate",
  description: string,        // verbatim from catalog HTML
  semester_offered: string | null,  // "F, Sp" etc., when published
  prereqs: Prereq | null,
  coreqs: Prereq | null,
  cross_listed: string[],     // ["ECE 4153"] when MSU notes equivalency
  source_url: string          // "https://catalog.msstate.edu/search/?P=CSE%204153"
}

Prereq = {
  required_courses: string[],         // lossless from prose
  logic: "or" | "and" | "mixed" | null,  // best-effort
  min_grade: "A" | "B" | "C" | "D" | null,  // best-effort
  non_course: string[],               // ["junior standing", "consent of instructor"]
  raw_prose: string                   // verbatim — escape valve for ambiguity
}
```

**`get_msu_course_graph`**

```ts
input: { code: string, direction: "prereqs" | "unlocks", depth?: number }
output: {
  root: string,
  direction: "prereqs" | "unlocks",
  depth_requested: number,
  depth_used: number,
  nodes: Array<{ code, title, depth }>,
  edges: Array<{ from, to, logic, min_grade }>,
  truncated: boolean,         // true if depth cap hit
  notes: string[]
}
```

## 4. Data flow

### Build time (`scripts/build-worker-corpus.mjs` → spawns `_scrape-catalog.ts`)

```
1. fetch https://catalog.msstate.edu/azindex/
2. extract every dept-page href matching CATALOG_ROOTS allowlist
3. for each dept page:
     fetch the page
     extract every bubblelink course code (validate against /^[A-Z]{2,4}\s\d{4}$/
       on the *visible link text*, not the attribute string)
4. dedupe course codes across all depts
5. for each unique code (concurrency 4, jitter 200–600 ms):
     fetch /search/?P=<DEPT>%20<NUM>
     parse → Course record (see § 4.1)
6. build forward_dag: { code: [prereq_codes...] }
7. build reverse_dag: invert forward_dag
8. abort if:
     - any HTTP 5xx after retries
     - any dept yields zero courses
     - any WAF interstitial detected (string match on known CourseLeaf
       challenge markup, mirroring policy-build pattern)
     - prereq-parse exception rate > 5% of corpus
     - reverse_dag is empty when forward_dag is non-empty
9. write worker/corpus.json.courses = { version, scraped_at, records, forward_dag, reverse_dag }
10. emit log line "course corpus: <N> records, <M> dag edges" to stderr
```

### 4.1 Prereq parser (two-pass)

Catalog prose example:
> `(Prerequisites: Grade of C or better in CSE 3724 or ECE 3724). Three hours lecture. The concepts and practices of data communications…`

**Pass 1 — high-recall course-code extraction.** Inside the parenthesized prereq sentence, run regex `\b[A-Z]{2,4}\s\d{4}\b`. Every match populates `required_courses`. Lossless — this is the field downstream graph-walking depends on.

**Pass 2 — best-effort logic + grade + non-course.**

- `logic`: presence of " or " between course codes → `"or"`; presence of " and " → `"and"`; both present → `"mixed"`; one course → `null`.
- `min_grade`: regex `Grade of ([ABCD])(?:\s+or\s+better)?` → letter.
- `non_course`: scan for phrases `"consent of instructor"`, `"permission of (instructor|department head)"`, `"(junior|senior|graduate)\s+standing"`, `"ACT \d+"`, `"SAT \d+"`. Each match → string appended to array.
- `raw_prose`: the full parenthesized prereq sentence, verbatim.

When pass 2 is uncertain (mixed booleans, unfamiliar phrasing), set `logic: "mixed"` and rely on `raw_prose` for human / LLM verification. The tool description for `get_msu_course` documents that `logic`, `min_grade`, and `non_course` are best-effort; `required_courses` and `raw_prose` are authoritative.

### Query time (mirrored across stdio and Worker)

`search_msu_courses(q, limit)`:
```
1. validate q (length ≤ MAX_QUERY_CHARS = 4096)
2. tokenize, BM25 rank over (code + title + description) index
3. return top-`limit` matches with diagnostic notes
```

`get_msu_course(code)`:
```
1. validate code via /^[A-Z]{2,4}\s\d{4}$/ after uppercase + trim
2. lookup in records map
3. hit → return course
   miss → return { found: false, suggestions: top-3 BM25 matches on code }
```

`get_msu_course_graph(code, direction, depth=5)`:
```
1. validate code; clamp depth to [1, 10]
2. select forward_dag or reverse_dag based on direction
3. BFS from code with visited-set cycle detection
4. stop when depth_used == clamped_depth OR queue empty
5. build nodes[] + edges[]; set truncated = (depth_used == cap && queue non-empty)
6. return graph; include notes[] for cycle detections and clamps
```

## 5. Error handling & graceful degradation

### F1 — Course not in corpus

`get_msu_course("FAKE 9999")` → `{ found: false, code: "FAKE 9999", suggestions: [...top-3 BM25 matches on code...] }`. Never throws; never echoes internal state. Generic message; no leak of corpus contents or file paths.

### F2 — Malformed course code

Fails regex validation `/^[A-Z]{2,4}\s\d{4}$/` after uppercasing + trimming whitespace and normalizing internal spaces. Reject early with `{ error: "invalid_course_code" }`. No code path that constructs catalog URLs from unvalidated input ever runs.

### F3 — Graph cycle hit during walk

Detected by visited-set. Walker stops at first re-encounter, appends `"cycle detected at <NODE>"` to `notes`, sets `truncated: true`. Result remains usable (partial graph).

### F4 — Depth out of range

`depth` arg clamped silently to `[1, 10]`. Result includes `depth_requested` and `depth_used`; client can compare. Log clamp to stderr only.

### F5 — Worker error paths

Never echo `(err as Error).message` to clients. Log structured fields server-side; return generic message with JSON-RPC `id`. Same invariant as round-2 closure for policy/calendar tools.

### F6 — Build-time scrape failures

Abort with marker string `"refusing to ship a poisoned course corpus"`. Specific abort triggers listed in § 4. Mirrors v0.4.0 calendar-build defense; no weakening.

### F7 — Argument size

Inputs > `MAX_QUERY_CHARS` (4096) → reject in tool handler before parse. Worker's existing `Content-Length > 64_000 → 413` body cap is the outer gate; tool-arg cap is the inner gate.

## 6. Security checklist additions

Adds CAT1–CAT4 to `tools/security-checklist.sh`, parallel to v0.4.0's CAL1–CAL4 for calendars. CI gate stays at `score >= 100`; current target moves from **192 → ~202**. Each grep is small and brittle by design — a regression must be deliberate and visible.

| ID | Check | Why |
|----|-------|-----|
| **CAT1** | Every `https://` URL inside `msstate-policies/src/courses/` stays on a `msstate.edu` subdomain (grep). | Corpus rule — no third-party hosts. |
| **CAT2** | Worker handler for each of `search_msu_courses`, `get_msu_course`, `get_msu_course_graph` caps `code.length` / `q.length` ≤ `MAX_QUERY_CHARS` before parsing. | Quadratic-regex / abuse defense. |
| **CAT3** | Build script contains literal `"refusing to ship a poisoned course corpus"` and the abort branch is reachable from every documented failure mode in § 4 / § 5. | Prevents silent corpus drift. |
| **CAT4** | `msstate-policies/src/courses/types.ts` defines a frozen `CATALOG_ROOTS` allowlist; no `new URL(`https://${...}`)` construction in the courses module reaches `https://` from a non-constant string. | Prevents URL-template injection. |

Round-3 closure note in `docs/BUILD.md` captures the per-check intent so a future maintainer can read the score target and understand which checks gate what.

## 7. Testing & eval strategy

### Unit tests

Five files under `msstate-policies/tests/courses/` covering parser fixtures, scraper enumeration, DAG-walk correctness, end-to-end corpus loading, and MCP tool handler shapes (see § 3 table). Vitest, same harness as existing tests.

### Eval set (`evals/courses.jsonl`)

~50 hand-curated questions, **every** `expected_codes` entry verified against a live `catalog.msstate.edu` fetch in the same session. Three buckets:

```jsonc
// course_explain: ~20 queries
{ "type": "course_explain",
  "q": "what's MSU's networking class?",
  "expected_codes": ["CSE 4153"] }

// prereq_chain: ~15 queries
{ "type": "prereq_chain",
  "q": "what do I need before CSE 4733?",
  "expected_codes_subset": ["CSE 3724", "CSE 1384", "CSE 1284"] }

// unlocks: ~15 queries
{ "type": "unlocks",
  "q": "what unlocks after CSE 1384?",
  "expected_codes_subset": ["CSE 2383", "CSE 2213", "CSE 3724"] }
```

`expected_codes_subset` semantics: returned set must be a **superset** of the expected subset. Lets the eval tolerate MSU adding new courses that legitimately reference the prereq, without false-failing the eval.

### Eval runner

Reuses `scripts/run-evals.mjs`. New `--suite=courses` flag. CI runs the courses suite on any PR touching `src/courses/`, `worker/src/`, or `evals/courses.jsonl`. **Eval cost: zero.** No LLM, no embedding API, no external calls. Pure tool-output set-membership assertions.

### Manual pre-release smoke

1. `node msstate-policies/dist/index.js` — wire to MCP inspector, exercise all three tools.
2. `npx -y msstate-policies-mcp@<rc>` — same exercise.
3. Plugin install via `/plugin install msstate-policies@msstate-mcp` in Claude Code — same exercise.
4. `wrangler deploy` to a staging Worker; `curl` JSON-RPC against each new tool.

## 8. Cost analysis

| Vector | Build time | Runtime |
|--------|------------|---------|
| Catalog HTTP fetches (build only) | ~1,500–2,000 polite GETs to `catalog.msstate.edu`, free | Zero — no runtime fetch surface |
| Corpus storage | +~400 KB gzipped in `worker/corpus.json`; under CF Worker bundle limit | Free |
| BM25 search | n/a | In-process; zero external calls |
| LLM / embedding APIs | None | None |
| Cloudflare Worker invocations | n/a | Existing free-tier headroom (~10/day current) |

**Conclusion: zero ongoing operational cost.** The only spend is developer time during builds. Embeddings, if eval ever justifies them, are an additive non-breaking change (see § 9).

## 9. Open questions & deferred items

| ID | Item | Disposition |
|----|------|-------------|
| **D1** | Embeddings for course search | Deferred. BM25 over `code + title + description` is strong on this corpus. Three viable upgrade paths if v0.6.0 eval flags semantic-gap failures: (a) reuse v0.5.0 Voyage integration (free tier covers it), (b) Cloudflare Workers AI (`@cf/baai/bge-small-en-v1.5`, 10k neurons/day free), (c) dept-alias dictionary (`{"networks": ["CSE 4153"], ...}`). Decision deferred to a v0.6.1 if needed. |
| **D2** | Degree-program structure (`get_msu_program`) | Deferred to v0.7. `sc_courselist` + `orclass` parsing is meaningful additional surface area. |
| **D3** | Archived catalog editions | Deferred. ~3× corpus size; useful for "I'm on the 2023–24 catalog" cohort but not v1. |
| **D4** | Cross-listed course de-duplication | Best-effort in v1. If catalog says `CSE 4153` and `ECE 4153` are the same class, we store both records with `cross_listed: ["<other>"]` and let the LLM client surface the equivalence. A v0.6.1 follow-up could canonicalize. |
| **D5** | Co-req vs. pre-req disambiguation | Parser captures both into separate `Prereq` / `Coreq` blocks; graph walk uses `Prereq` only by default. A future `include_coreqs` flag on `get_msu_course_graph` would extend this. |
| **D6** | Course offerings (which semester / which professor) | Out of corpus scope. Belongs to a separate registrar-BANNER integration, not catalog.msstate.edu. |
| **D7** | Refresh cadence | Catalog editions ship ~yearly (summer). A v0.6.x release pinned to each new edition is the expected operating model. No cron; no scheduled re-scrape. |

## 11. Decisions captured during brainstorm

- **Primary user:** all three (students, prospective, advisors), optimize for current students.
- **v1 pains in scope:** prereq chain (forward), course explainer, reverse "unlock" lookup. **Deferred:** degree-program structure.
- **Prereq fidelity model:** hybrid (lossless `required_courses` + lossless `raw_prose` + best-effort `logic` / `min_grade` / `non_course`).
- **Catalog slice:** current undergrad + grad, no archives.
- **Chain walking:** single tool call returns full DAG (depth-capped); no per-hop tool calls.
- **Delivery:** baked corpus, no runtime catalog fetches.
- **Search:** BM25 only for v0.6.0; embeddings deferred (D1).
- **Cost ceiling:** zero ongoing operational cost is a hard requirement, not a target.
