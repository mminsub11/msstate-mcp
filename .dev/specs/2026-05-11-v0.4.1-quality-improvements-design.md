# v0.4.1 — Service quality improvements (cheap-wins release)

**Date:** 2026-05-11
**Status:** Design (awaiting implementation plan)
**Scope:** Five server-side changes that reduce calendar response noise, lock in URL citation, and surface cross-source coverage gaps to the LLM — without yet adding semantic-embedding infrastructure (which is deferred to v0.5.0).

## Goal

Reproducibly fix two user-reported quality issues in the v0.4.0 calendar tools:

1. **Intermittent URL surfacing** — the LLM sometimes drops `source_url` when summarizing tool output. Fix: pre-format a markdown citation link in every row so the LLM can copy it verbatim, and tighten the tool description to require it.

2. **Cross-calendar context gaps** — when a per-audience source (grad school, financial aid, housing) has no data for a queried term, the LLM concludes "no data" rather than falling back to the authoritative academic calendar. Fix: server-side smart fallback that adds top academic-calendar matches for any term mentioned in the query, tagged so the LLM can present them honestly.

Plus a corpus-quality fix that drops ~22% duplicate rows discovered during root-cause investigation.

## Non-goals

This is the cheap-wins release. Out of scope:

- **Semantic embedding for calendar rows** and hybrid BM25 + embed retrieval. The root-cause investigation confirmed BM25 misses natural-language phrasings like *"when does the semester start"* against terse event titles like *"Classes begin"*. The right fix is semantic embedding (matching the policy path's `dist/embeddings.json` pipeline), but it's a separate ~3-day project that ships as v0.5.0.
- **Cross-source canonical-row promotion** — when "Classes begin Spring 2026" appears in academic, grad, and SFA, we keep all three. Promoting one as canonical and demoting others is deferred (would require source-authority metadata richer than the parent-pointer we add here).
- **New tools, new env vars, new MCP surfaces.** Tool count stays at 7. Worker URL unchanged. Connector flow unchanged from the user's perspective.
- **`coverage` field on response** — exposing per-source row counts so the LLM can reason about gaps. The smart fallback handles the most common case (term-anchored queries) without needing this metadata. Defer unless v0.4.1 in production shows the LLM still hitting gaps the fallback doesn't catch.
- **Tool selection nudges via description** — root-cause investigation showed BM25 ranking is the bigger issue than which tool the LLM picks. Tool description changes in this round are about output formatting + fallback handling, not which tool to call.
- **Version bump strategy** beyond patch (v0.4.0 → v0.4.1). No major or minor surface change.

## Background — what the root-cause experiments showed

The investigation that motivates this spec found:

- **BM25 ranking is semantically thin.** Query *"when does spring 2027 start"* returns 10 matches with zero relevant rows; top hits are graduation-application deadlines that happen to mention "Spring 2027". Query *"classes begin spring 2027"* returns the correct row at rank 1 with score 23.18. The corpus has the answer; BM25 can't find it from natural-language phrasing. → addressed by v0.5.0 embeddings; this round does what we can without that.
- **Corpus has 22% duplicate rows** (201 of 914). academic_calendar is the worst offender (125/435 = 29% dup rate); grad_school is next (38/146 = 26%). Per-term sub-pages overlap on the source side, and the parser re-emits the same event under multiple term labels. → addressed by C1 below.
- **All source URLs return 200 OK.** Rules out a "dead URL" hypothesis for the URL problem. → addressed by C2/C5: better LLM rendering, not URL repair.
- **Cross-source asymmetry is real.** Spring 2027 exists in academic_calendar but not yet in grad_school_calendar (MSU hasn't published the grad PDF for 2027). The LLM has no signal that "your source is missing this term but the parent has it." → addressed by C3/C4 fallback path.

## Architecture

Five independent, additive server-side changes. No restructuring of existing modules; this is incremental on top of v0.4.0's parser + scraper + search + tool wiring.

### C1. Within-source dedup (parser fix)

Each parser deduplicates identical `(event, start)` tuples before emitting. Implemented inside each extractor as:

```typescript
const seenKey = new Set<string>();
// ...inside the row-emit loop:
const key = `${event}|${start}`;
if (seenKey.has(key)) continue;
seenKey.add(key);
```

Affects four files:
- `msstate-policies/src/calendars/parsers/date_table.ts` (Shape A — holidays)
- `msstate-policies/src/calendars/parsers/term_pages.ts` (Shape B — registrar academic + exam + SFA)
- `msstate-policies/src/calendars/parsers/event_list.ts` (Shape C — housing)
- `msstate-policies/src/calendars/parsers/pdf_calendar.ts` (Shape D — grad school)

Dedup is **scoped per parser invocation** — running the SFA parser on the Fall 2026 sub-page produces unique rows for that sub-page; the same event landing in a separate Spring 2026 sub-page produces a separate emit. Aggregation across sub-pages is the scraper's responsibility (in `scraper.ts`'s `scrapeTermB`), which also dedupes when it merges results from multiple term pages:

```typescript
// In scrapeTermB after collecting per-sub-page rows:
const aggregateSeen = new Set<string>();
const dedupedRows = rows.filter((r) => {
  const k = `${r.source}|${r.event}|${r.start}|${r.term ?? ""}`;
  if (aggregateSeen.has(k)) return false;
  aggregateSeen.add(k);
  return true;
});
```

Aggregate-level dedup keys also include `term` because a single event can legitimately occur in two different terms (e.g., "Classes begin" in Spring 2027 AND Fall 2027 are different facts and must both survive).

Expected effect: **914 → ~715 rows.** Validation: after rebuild, the audit script in §Testing reports ≤ 5 duplicate event-date pairs across the whole corpus (down from 156).

### C2. `citation` field on every `CalendarRow`

Extend the type in `msstate-policies/src/calendars/types.ts`:

```typescript
export interface CalendarRow {
  source: CalendarSource;
  event: string;
  start: string;
  end: string;
  time?: string;
  term?: string;
  description?: string;
  source_url: string;
  retrieved_at: string;
  /** Pre-formatted markdown link for the LLM to include verbatim in its answer. */
  citation: string;
  /** Set to true by `find_msu_date` when this row was appended via the smart-fallback
   *  path (C4). Never set at parse/scrape time; never serialized into worker/corpus.json. */
  fallback?: boolean;
}
```

`citation` is computed at scrape time and serialized into `worker/corpus.json`. `fallback` is added at runtime by the chain tool only — it never appears in the static snapshot.

Compute the citation at parser time inside the `CalendarRow` constructor helper (or inline at each row emission, since parsers all build `CalendarRow` objects directly today):

```typescript
function formatCitation(event: string, term: string | undefined, source_url: string): string {
  const label = term ? `${event}, ${term}` : event;
  // Trim labels longer than 80 chars for readability in chat.
  const safe = label.length > 80 ? label.slice(0, 77) + "…" : label;
  return `[${safe}](${source_url})`;
}
```

Examples:
- `[Spring Break, Spring 2026](https://www.registrar.msstate.edu/calendars/academic-calendar/2026/spring)`
- `[Independence Day Holiday](https://www.hrm.msstate.edu/benefits/holidays/)` (no term for holidays)
- `[Halls Close for Spring 2026](https://www.housing.msstate.edu/events/)` (no term for housing events)

The format is the same for all 6 sources. Both Claude and ChatGPT render markdown links cleanly; this is the simplest format that works across clients.


Used only by C4 (smart fallback). Exposing this through any tool or LLM-visible field is out of scope this round — keep it as an internal invariant for the chain tool's fallback logic.

### C4. Smart fallback in `find_msu_date`

The chain tool's behavior expands. After the existing BM25 search step:

```
matches = bm25SearchCalendars(q, 10)            # existing behavior

# New: detect a term in the query
term = matchTerm(q)                              # /(Spring|Fall|Summer|Winter|Maymester)\s+\d{4}/i
if term is not None:
    academic_count = count(m for m in matches if m.source == "academic_calendar")
    if academic_count < 2:
        # Run a targeted second pass: academic_calendar rows with matching term
        extras = [r for r in CAL_ROWS
                  if r.source == "academic_calendar"
                  and r.term == term
                  and r.source_url not in [m.source_url for m in matches]]
        # Sort extras by a small heuristic — prefer "Classes begin", "Final Exam", etc.
        extras = sort_by_event_priority(extras)[:3]
        for e in extras:
            e.fallback = True
            matches.append(e)
```

Triggering conditions:
- Term reference matches the regex (case-insensitive)
- Fewer than 2 of the top-10 BM25 results are from `academic_calendar`

Fallback rows:
- Up to 3 academic_calendar rows for the exact-matched term
- Sorted by an internal priority (term-boundary events first: "Classes begin", "Last day of classes", "Final Exams", "Spring Break", "Thanksgiving"; then other dated events)
- Appended to the `matches` array (NOT interleaved into BM25 ranking)
- Each tagged with `fallback: true`

The `notes` field gets a new variant when fallback fires:

```
"Added academic_calendar rows for {term} — if your primary source didn't have this term, the academic calendar is authoritative for term-boundary dates."
```

Fallback runs only in `find_msu_date` (the chain tool). `get_msu_calendar` (the raw getter) stays deterministic — it returns exactly what was requested, no widening. Power users querying a specific source expect raw results.

### C5. Tool description tightening

`find_msu_date.description` gets two new rules and one rephrased rule. The full new description:

> Answer natural-language questions about Mississippi State University academic dates, financial-aid deadlines, university holidays, residence-life milestones, and graduate-school deadlines. Returns up to 10 matching calendar rows ranked by relevance, plus up to 3 academic-calendar fallback rows when a term is mentioned and the primary source has limited coverage.
>
> Each row has:
> - `start`, `end` — ISO dates (YYYY-MM-DD).
> - `event`, `term`, `description` — what happened and when.
> - `source`, `source_url` — which calendar and the canonical msstate.edu URL.
> - **`citation`** — a pre-formatted markdown link. **You MUST include this verbatim in your final answer for every row you reference. Do not paraphrase or omit it. Format: `[Event, Term](url)`.**
> - **`fallback`** (optional) — if `true`, this row came from the academic-calendar fallback because the user's primary source didn't have data for the mentioned term. Surface this explicitly to the user: *"Your primary source doesn't list this date for that term, but the academic calendar shows..."*
>
> RULES:
> 1. Use ONLY the returned rows. Do not invent dates or draw on outside knowledge.
> 2. Quote the date verbatim. Always include the `citation` field as a clickable link.
> 3. If the user's question does NOT specify a year/term and multiple year-versions of an event exist (e.g., Spring Break 2026 and Spring Break 2027), present ALL of them year-by-year with each one's citation.
> 4. If `matches` is empty or no row clearly answers the question, say so plainly; do not extrapolate.
> 5. Surface the row's `retrieved_at` and `corpus_built_at` when accuracy is at stake.

`get_msu_calendar.description` gets a shorter update:

> Return the raw rows for one MSU calendar source. `source` is one of: academic_calendar, exam_schedule, university_holidays, grad_school_calendar, sfa_financial_aid, housing. Optional `term` filter matches via case-insensitive substring (e.g. 'Fall 2026', '2026', 'fall'). Each row has a pre-formatted `citation` markdown link — include it verbatim when surfacing any specific date to the user.

Both descriptions are mirrored in `worker/src/index.ts`'s `TOOLS` array (Worker duplicates descriptions; this is the existing pattern).

## Data flow

```
Build time:
  scrape (live MSU fetch)
    → parsers (now: dedupe within-source + compute citation)
    → scraper.ts aggregation (now: dedupe at term-page-merge boundary)
    → corpus rows
    → worker/corpus.json (with `citation` on every row)

Local install runtime (live scrape):
  same pipeline; in-memory TTL cache (6h housing, 24h others)

Worker runtime (snapshot):
  corpus.json loaded at module init; CAL_ROWS array used by find_msu_date + get_msu_calendar

Request: find_msu_date(q)
  → bm25SearchCalendars(q, 10) [unchanged ranking]
  → matchTerm(q) → optional term
  → if term and <2 academic matches in top 10:
       → filter CAL_ROWS for source=academic_calendar AND term=match
       → priority-sort top 3
       → append with fallback=true
  → response: { q, matches: [...regular, ...fallback], notes, corpus_built_at }
                each match has citation; some have fallback: true

Request: get_msu_calendar(source, term?)
  → filter CAL_ROWS by source [+ optional term substring]
  → response: { source, term, rows, source_url, corpus_built_at }
                each row has citation; no fallback flag
```

## Error handling

No new failure modes:

| Failure | Behavior |
|---|---|
| Dedup-key collision on legitimate-distinct rows | Cannot happen: keys are `event + start [+ term + source]` which uniquely identify a fact. |
| `citation` computation throws | The function is pure string-concat with no edge cases; reachable only if `source_url` is missing, which is validated upstream by the parser already. |
| Term-regex fails to match | Falls through; no fallback; behaves exactly like v0.4.0. |
| Fallback academic_calendar query returns 0 rows | Adds zero rows quietly; response shape unchanged; no error. |
| LLM ignores the `citation` rule | This is by design out of our control; the tool description nudges but doesn't enforce. Worst case: same behavior as v0.4.0. |

Security checks (CAL1–CAL4 in `tools/security-checklist.sh`) remain unchanged. No new URLs in `CALENDAR_URLS`. No new fetch surfaces. Query-length cap (`MAX_QUERY_CHARS`) and body-size cap (64 KB) unchanged on the Worker side.

## Security

No new security surfaces. The five changes:

- C1 (dedup): pure data-shape change; if anything, slightly reduces the corpus payload size.
- C2 (citation field): adds a derived string field per row; no new fetch, no new URL.
- C3 (parent metadata): a static constant; no runtime input.
- C4 (smart fallback): a server-side filter over already-fetched corpus; no new fetch.
- C5 (description text): LLM-side guidance; no server-side behavior change beyond what C2/C4 already added.

`tools/security-checklist.sh` continues to score 192/192. The CAL1–CAL4 checks all keep passing:

- CAL1 (URLs hardcoded in types.ts): unchanged.
- CAL2 (calendar code stays on msstate.edu): unchanged.
- CAL3 (Worker caps `find_msu_date` q length): unchanged.
- CAL4 (build aborts on WAF/empty): unchanged.

## Risks

**R1. Dedup keys are too narrow and lose legitimate-distinct rows.**
Likelihood: low. The within-source key `event|start` is what users would call "the same fact"; the aggregate key adds `term` so cross-term recurrences survive. Mitigation: regression eval catches this if a real loss happens.

**R2. The fallback fires too aggressively and adds noise.**
Likelihood: medium. The trigger (`<2 academic_calendar matches in top 10`) might fire on queries where the user genuinely wants the per-audience source. Mitigation: the `fallback: true` flag lets the LLM downweight these rows in its answer; the `notes` text frames them as supplemental. Tune threshold in v0.5.0 if data warrants.

**R3. LLM still drops `citation` despite the description.**
Likelihood: medium. The pre-formatted link should help, but LLMs vary in description adherence. Mitigation: post-launch, sample real conversations to measure citation-inclusion rate. If still inconsistent, escalate to "wrap the entire response as a tool-call output that requires citation" — but that's a v0.5.0+ design.

**R4. `citation` makes responses too verbose for terse queries.**
Likelihood: low. Markdown links render compactly in both Claude and ChatGPT; the user sees `Spring Break` as a clickable phrase rather than a raw URL. Worst case: slightly more visual chrome on simple answers.

**R5. Dedup reveals data-coverage gaps that were previously hidden by noise.**
Likelihood: certain (it's the point). After dedup, the corpus is ~715 rows instead of 914; some queries that returned 10 noisy matches will now return 5-6 cleaner matches. Mitigation: smart fallback (C4) fills the gap when it can; otherwise the LLM honestly says "no data" instead of fabricating from noise.
